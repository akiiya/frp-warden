# 0006 — 配置加载与自动生成

- 状态:已接受
- 日期:2026-06-12

## 背景

Phase 1 需要让配置系统真正落地:从 YAML 文件加载配置、支持命令行指定路径、首次启动
零配置可用、并对配置做基础校验。这些选择会影响部署体验与安全性,值得记录。

## 决策

### 为什么使用 YAML

沿用 [0004](0004-use-yaml-config.md):YAML 易读、便于手工编辑、支持注释与层级结构,
适合面向运维人员的配置。解析使用成熟轻量的 `gopkg.in/yaml.v3`,不引入额外重型依赖。
`internal/config` 中的 Go struct(及其 yaml tag)是配置 schema 的唯一来源,
[../CONFIGURATION.md](../CONFIGURATION.md) 必须与之同步。

### 为什么默认零配置启动

目标是"运行单一二进制即可用"。因此:配置文件不存在时自动写入默认配置并继续运行;
文件中缺省的键沿用代码内默认值;使用 SQLite 时自动创建数据目录。运维无需先手写配置
即可启动,降低上手成本。

同时坚持"不静默忽略错误":配置文件**格式错误**或**校验不通过**时,程序明确报错并退出,
而不是带着错误配置继续运行。

### 为什么 session_secret 首次启动自动生成

`session_secret` 用于会话/签名,必须是高熵且每个部署唯一的值。绝不能写死默认值
(否则所有部署共享同一密钥,可被伪造会话)。因此当其为空时,首次启动用 `crypto/rand`
生成至少 32 字节熵的随机值(base64 编码)并写回配置文件;严禁使用 `math/rand`。
含密钥的配置文件以 `0600` 权限写入,视为敏感文件。

### 为什么 plugin_addr 默认只监听 127.0.0.1

plugin 端点参与 frps 的授权决策,是安全关键面。默认仅绑定回环地址,避免因误配置而
暴露到公网导致授权被外部影响(见 [../SECURITY.md](../SECURITY.md) 与
[0002](0002-use-frps-http-plugin.md))。需要跨主机时应走可信私有链路,而非公网绑定。

### 为什么本轮暂不引入数据库驱动

Phase 1 聚焦配置系统,刻意保持范围最小。数据目录初始化只创建目录、不连接数据库;
SQLite 驱动选型(纯 Go、无 CGO,见 [0005](0005-use-sqlite-by-default.md))与迁移留到
Phase 2。这样可避免过早引入依赖,保持交叉编译简单,也让每个阶段可独立评审。

## 影响

- 首次启动会在工作目录生成 `config.yaml` 与数据目录(默认 `./data`)。
- 配置 struct 与 [../CONFIGURATION.md](../CONFIGURATION.md) 必须始终同步。
- 自动写回会以程序序列化结果重写文件(顶部附说明注释),用户手写的额外注释可能丢失。
- 数据库相关能力(连接、迁移、初始管理员)依赖 Phase 2,本阶段尚不可用。
