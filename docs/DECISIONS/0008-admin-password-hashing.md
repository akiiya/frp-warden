# 0008 — 默认管理员密码:随机生成与 bcrypt 哈希

- 状态:已接受
- 日期:2026-06-12

## 背景

frp-warden 的管理后台必须登录才能管理。系统需要一个初始管理员账号,但绝不能使用
硬编码的弱默认凭据(如 `admin/admin`)。同时,密码的存储方式直接关系到系统安全。

## 决策

- 首次启动时,若 `admins` 表为空,自动创建默认管理员:用户名取
  `security.initial_admin_username`,**密码强随机生成**(`crypto/rand`,长度落在 16–24,
  默认 20,剔除易混淆字符)。
- 该明文密码**只在创建时一次性打印**到控制台,绝不持久化、绝不写入配置文件或日志文件。
- 密码**只以 bcrypt 哈希存储**(`bcrypt.DefaultCost`),封装在 `internal/security`。
- 新建管理员的 `must_change_password` 默认为 `true`。
- 若已存在任意管理员,则不创建、不打印密码。

## 为什么这样选择

- **为什么默认密码随机生成。** 杜绝弱默认凭据被批量利用;每个部署的初始密码唯一、高熵。
- **为什么密码只显示一次。** 明文不落盘是最稳妥的做法——只在创建瞬间于控制台呈现,
  运维记录后即不可再获取,降低明文泄露面。
- **为什么使用 bcrypt。** bcrypt 成熟、自带 salt、可调 cost,Go 生态支持完善
  (`golang.org/x/crypto/bcrypt`),足以满足当前阶段需求;`DefaultCost` 是合理默认值。
- **为什么 `must_change_password` 默认为 true。** 初始密码由系统生成并经控制台传递,
  应被视为临时凭据,强制首次登录后修改可缩短其有效窗口。
- **为什么不保存明文密码。** 明文一旦落库或入日志即构成长期泄露风险;只存不可逆哈希
  可在数据库泄露时显著降低危害。

## 预留空间(不过度设计)

`internal/security` 仅暴露 `GenerateRandomPassword` / `HashPassword` / `VerifyPassword`
三个简单函数。未来若需切换到 argon2id,只需替换实现而不影响调用方;本轮不提前引入
多算法选择逻辑。

## 影响

- 运维必须在首次启动时记录控制台打印的初始密码;丢失则需通过后续提供的重置手段处理。
- 登录与改密 API 属后续阶段(见 [../ROADMAP.md](../ROADMAP.md));本轮只打好初始化地基。
