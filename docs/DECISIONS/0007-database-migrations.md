# 0007 — 轻量内置数据库迁移框架

- 状态:已接受
- 日期:2026-06-12

## 背景

frp-warden 需要管理数据库 schema 的演进。我们希望保持单一二进制、零外部依赖、
跨平台易构建(无 CGO),并且不引入重型工具或框架。

## 决策

在 `internal/db` 中实现一个**轻量、内置**的迁移框架,基于标准库 `database/sql` +
手写 SQL:

- 维护一张 `schema_migrations` 表(`version`、`name`、`applied_at`)。
- 每条迁移有唯一、递增的版本号与名称,语句在单个事务中按序执行。
- 启动时确保 `schema_migrations` 存在,读取已应用版本,**按顺序只执行未应用的迁移**;
  成功后写入 `schema_migrations` 再提交。
- 任意一条迁移失败立即回滚并返回错误(**fail fast**),程序据此退出,绝不静默跳过。

## 为什么这样选择

- **为什么用轻量内置框架。** 迁移需求简单(顺序、只跑一次、可重复执行安全),内置实现
  仅几十行,易读、易审计、无第三方迁移工具的版本与兼容负担。
- **为什么本轮先完整支持 SQLite。** 项目默认 SQLite 且要求零配置可用;先把默认路径
  做扎实、做对(连接、迁移、测试覆盖),比同时铺开多数据库更可控。
- **为什么不引入 ORM。** ORM 带来抽象成本、反射开销与隐式行为;本项目数据访问简单,
  手写 SQL 更透明、更利于精确控制唯一约束等关键不变量(见 [../DATA_MODEL.md](../DATA_MODEL.md))。
- **为什么 database/sql 足够当前阶段。** 标准库已提供连接池、事务、参数化查询,
  满足当前所有需求,且不增加依赖面。

## 其它数据库当前状态

- **SQLite**:完整可用(纯 Go 驱动 `modernc.org/sqlite`,无 CGO),迁移已实现并测试覆盖。
- **MySQL / MariaDB**(`github.com/go-sql-driver/mysql`)与 **PostgreSQL**
  (`github.com/jackc/pgx/v5/stdlib`):已注册驱动、可建立连接,但**迁移尚未实现**。
  调用迁移时会 fail fast 报错,绝不写"看似支持实则不支持"的半成品。不同数据库 SQL 语法
  存在差异,其迁移留待后续阶段按方言实现。

## 影响

- 新增迁移时只能**追加**新版本号,不得修改既有迁移(其是否执行由 `schema_migrations` 决定)。
- 多数据库支持需要为各方言分别提供迁移 SQL;在此之前,生产部署应使用 SQLite。
- 迁移失败会阻止启动,这是刻意的:宁可不启动,也不在错误的 schema 上运行。
